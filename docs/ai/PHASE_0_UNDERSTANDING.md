## 1. 项目目标

### 1.1 核心目标
将现有 `academic-agent-team` 项目升级为 **PaperGenius Pro** — 一个集成 20+ GitHub 开源项目的模块化端到端学术写作平台。

### 1.2 具体交付物
| 交付物 | 描述 | 优先级 |
|--------|------|--------|
| PDF 解析 + RAG | 上传 PDF → 结构化提取 → 向量存储 → 问答检索 | P0 MVP |
| 文献自动搜索 | arXiv / Semantic Scholar / CNKI 批量检索 | P0 MVP |
| 向量数据库集成 | Chroma 本地持久化文献知识库 | P0 MVP |
| 数据分析沙箱 | Python 代码安全执行 + 图表生成 | P1 |
| 引用管理 | BibTeX 提取 + GB/T 7714 / APA / IEEE 格式化 | P1 |
| LaTeX 编译 | 本地编译 + 中文核心期刊模板 | P1 |
| 查重/降重 | 语义相似度检测 + 改写建议 | P2 |
| 投稿助手 | 期刊匹配 + Cover Letter 生成 | P2 |
| 插件市场 | 一键安装 GitHub 项目作为新 Agent | P3 |

### 1.3 成功标准
- 输入关键词 → 输出可投稿论文（含真实引用）
- 生成时间 ≤ 15 分钟（云端）/ ≤ 30 分钟（本地）
- 查重通过率 ≥ 95%
- 支持本地全离线运行（Ollama）

## 2. 复杂性来源

### 2.1 技术复杂性
| 来源 | 风险等级 | 说明 |
|------|----------|------|
| PDF 解析精度 | 🔴 高 | 学术 PDF 含复杂公式、表格、多栏布局，解析失败率高 |
| RAG 召回质量 | 🔴 高 | 向量检索需 fine-tune embedding，否则幻觉严重 |
| 多 Agent 编排 | 🟡 中 | 现有 AutoGen 架构稳定，但新增 Tool Calling 需适配 |
| 代码沙箱安全 | 🟡 中 | 执行用户上传数据的代码存在注入风险 |
| 文献 API 限流 | 🟡 中 | arXiv/Semantic Scholar 有请求限制 |

### 2.2 产品复杂性
| 来源 | 风险等级 | 说明 |
|------|----------|------|
| 中英双语适配 | 🟡 中 | 中文核心期刊格式与英文顶刊差异大 |
| 学术诚信边界 | 🔴 高 | 需明确 AI 生成声明、引用率门槛 |
| 用户期望管理 | 🟡 中 | "一键生成论文" 易引发过高期望 |

### 2.3 集成复杂性
| 来源 | 风险等级 | 说明 |
|------|----------|------|
| 20+ GitHub 项目版本 | 🔴 高 | 依赖版本冲突、API 变更风险 |
| AutoGen vs LangGraph | 🟡 中 | Grok PRD 建议 LangGraph，现有项目用 AutoGen |
| 本地 vs 云端混合 | 🟡 中 | Ollama/Chroma 本地部署需额外配置 |

## 3. 若无约束会失败的方式

### 3.1 范围蔓延失败
> **场景**: 试图同时实现 Grok PRD 全部 20+ 功能，导致：
> - 6 个月无可用交付物
> - 技术债务累积
> - 团队精力分散

### 3.2 幻觉论文失败
> **场景**: RAG 质量不过关，生成论文包含虚假引用：
> - 用户投稿被退稿/学术不端
> - 产品信誉崩塌

### 3.3 架构腐化失败
> **场景**: 为赶进度绕过已批准架构，直接硬编码：
> - 插件无法热加载
> - 多 Provider 支持破坏
> - 测试无法覆盖

### 3.4 门禁虚设失败
> **场景**: 门禁确认流于形式，人类未仔细审核：
> - 带着重大设计缺陷进入编码
> - 后期发现问题需全面重构

## 5. 升级给人类的问题

### Q1: 编排框架选择
> **问题**: Grok PRD 建议使用 LangGraph，但现有项目已用 AutoGen GraphFlow。
> 
> **选项**:
> - A) 保留 AutoGen（最小改动，但与 Grok 生态不一致）
> - B) 迁移 LangGraph（大量重构，但与 Grok PRD 一致）
> - C) 双栈支持（复杂度高，维护成本大）
>
> **建议**: A（保留 AutoGen），理由是现有架构稳定，Tool Calling 可满足需求
